# AutoCertX 开源组件选型与扩展性设计 V1.0

- 编写日期：2026-04-15
- 适用范围：首个 GA 版本技术选型、组件复用边界、后续扩展性约束
- 关联文档：
  - `doc/需求说明书.md`
  - `doc/GA一期与后续需求规划.md`
  - `LICENSE`

## 1. 背景与目标

当前一期 GA 已冻结为以下范围：

- CA：Let's Encrypt
- 协议：ACME v2
- challenge：HTTP-01、DNS-01（TXT）
- DNS Provider：阿里云 DNS
- 部署目标：NGINX、Tomcat（仅 `JSSE + PKCS12`）
- 算法：一期仅 `RSA`
- 执行模型：客户侧多 Agent

同时，系统在后续版本必须具备以下扩展能力：

- 支持 `SM2` 等国密算法
- 支持标准 `ACME` 兼容前提下的私有扩展字段与私有流程
- 支持更多 CA、更多 DNS Provider、更多目标连接器
- 在扩展过程中不替换核心组件，不推倒已有状态机和资产模型

因此，本项目的选型原则不是“找到一个能跑通一期的全栈开源项目”，而是“构建一个以自研核心为主、以开源库为补充的可持续演进架构”。

## 2. 设计约束

### 2.1 核心约束

- 核心域模型必须自研，不能直接绑定某个 ACME 客户端或某个开源 CLM 产品的数据结构。
- ACME 只能作为 `IssuerDriver` 的一种协议实现，不能成为系统架构中心。
- 一期实现必须服务于后续 `SM2 + 私有 ACME 扩展`，不能采用后续必然重写的临时组件。
- 控制面和执行面的边界必须稳定：控制面编排、执行面持钥与部署。

### 2.2 许可证约束

当前仓库许可证文件为 `GNU LGPL v2.1`。在未调整主许可证前，工程上应优先选择以下许可类型的依赖：

- `MIT`
- `BSD / BSD-3-Clause`
- `ISC`

对于 `Apache-2.0` 组件，应谨慎使用，避免将其放入核心依赖路径；如后续确有必要，应先完成许可证策略评估再引入。

### 2.3 Go 版本约束

项目不再以 `Go 1.23` 为约束基线。为了兼容当前 ACME、国密和相关生态，建议直接升级到：

- 推荐基线：`Go 1.26.2`
- 最低建议：`Go 1.25.x`

原因：

- `lego` 当前主线依赖 `Go 1.25.0`
- `CertMagic` / `acmez` 当前依赖 `Go 1.24.0`
- `github.com/emmansun/gmsm` 新版本要求 `Go 1.24+`

## 3. 总体技术路线

### 3.1 路线结论

采用如下总体路线：

- 核心域模型自研
- 协议层自研封装
- 连接器/算法/协议驱动可插拔
- 复用若干聚焦型开源库，而不是引入一个全栈产品作为底座

### 3.2 分层模型

#### Core Domain

负责以下核心对象和状态机：

- 证书申请单
- challenge 跟踪
- 签发任务
- 部署任务
- 续期任务
- 发现记录
- 证书资产台账
- 审计事件

该层不得引用第三方 ACME 客户端的原生模型。

#### Protocol Layer

定义内部协议子系统，尤其是：

- `pkg/protocol/acme`

该层负责：

- 标准 ACME 请求/响应封装
- JWS/JWK/Nonce/Directory 处理
- account / order / authorization / challenge / finalize / download 流程
- 私有扩展字段与私有 challenge 流程的兼容承载

后续如需对 ACME 增加 `SM2` 扩展、私有扩展字段、私有 finalize 流程，应只在该层实现，不外溢到业务层。

#### Driver Layer

对外暴露稳定接口：

- `IssuerDriver`
- `KeyAlgorithmProvider`
- `ChallengeProvider`
- `DNSProvider`
- `TargetConnector`
- `DiscoveryProvider`

#### Runtime Layer

包括：

- 控制面 API / 调度 / 作业中心 / 审计
- Agent 注册 / 心跳 / 任务拉取 / 本地执行

## 4. 核心接口边界

### 4.1 IssuerDriver

职责：

- 查询 CA 能力
- 创建订单/签发流程
- 提交 challenge 结果
- finalize 并下载证书
- 续期与吊销

建议接口能力：

- `CreateOrder`
- `GetChallenges`
- `NotifyChallengeReady`
- `FinalizeOrder`
- `DownloadCertificate`
- `RenewCertificate`
- `RevokeCertificate`
- `GetCapabilities`

要求：

- `acme` 只是一个实现
- 后续 `vault-pki`、`custom-ca` 必须可平滑接入

### 4.2 KeyAlgorithmProvider

职责：

- 生成私钥
- 生成 CSR
- 输出证书链相关编码能力

一期实现：

- `rsa`

后续扩展：

- `sm2`
- `ecdsa`

要求：

- 业务层不得直接依赖 `crypto/rsa`
- 所有密钥与 CSR 流程必须走算法抽象层

### 4.3 ChallengeProvider / DNSProvider

职责：

- `HTTP-01` 文件落地
- `DNS-01` 记录写入、清理、查询

要求：

- `challenge` 类型与 `DNS provider` 解耦
- `DNS-01` 不能直接写死阿里云 SDK

### 4.4 TargetConnector

职责：

- 证书格式转换
- 目标文件写入
- reload/restart
- 部署后验证
- 回滚

一期实现：

- `nginx`
- `tomcat-jsse-pkcs12`

### 4.5 DiscoveryProvider

职责：

- 配置扫描
- 证书路径解析
- 本地证书元数据提取
- 有效期计算
- 发现结果回传

一期实现：

- `nginx-config`
- `tomcat-serverxml`

## 5. 开源组件选型结论

### 5.1 一期必选组件

#### 1. `golang.org/x/crypto/acme`

用途：

- 作为内部 `ACME` 协议子系统的一期基础实现

选型结论：

- 强烈推荐

原因：

- Go 官方维护
- 许可证友好
- 抽象层级足够低，适合做二次封装
- 不会把业务模型绑死在外部库结构上

使用原则：

- 只能在 `pkg/protocol/acme` 内部使用
- 业务层和调度层不得直接引用

#### 2. `github.com/libdns/libdns`

用途：

- DNS Provider 抽象层

选型结论：

- 强烈推荐

原因：

- 可将 `DNS-01` 与具体云厂商解耦
- 后续增加腾讯云、华为云、Route53 不需要改 challenge 编排逻辑

#### 3. `github.com/libdns/alidns`

用途：

- 阿里云 DNS Provider 实现

选型结论：

- 强烈推荐

原因：

- 一期范围明确只支持阿里云 DNS
- 与 `libdns` 配合使用，能把 provider 隔离在驱动层

#### 4. `github.com/tufanbarisyildirim/gonginx`

用途：

- NGINX 配置解析与发现

选型结论：

- 推荐

原因：

- 适合扫描 `ssl_certificate`、`ssl_certificate_key`、`server_name`
- 适合做发现链路，不要求自己维护 NGINX 配置语法解析器

使用限制：

- 主要用于发现和静态解析
- NGINX 配置写回、reload、回滚逻辑建议自研

#### 5. `software.sslmate.com/src/go-pkcs12`

用途：

- Tomcat `PKCS12` 编码/转换

选型结论：

- 强烈推荐

原因：

- 一期 Tomcat 范围就是 `JSSE + PKCS12`
- 库功能聚焦，不会把 Tomcat 连接器做重

### 5.2 后续预埋组件

#### 6. `github.com/emmansun/gmsm`

用途：

- 后续 `SM2 / SM3 / SM4 / 国密 x509` 能力实现

选型结论：

- 强烈推荐作为二期算法扩展基座

原因：

- MIT 许可
- 活跃维护
- 适合作为 `KeyAlgorithmProvider/sm2` 与后续 `smx509` 扩展的实现基础

要求：

- 一期先不启用 `SM2` 签发
- 但算法抽象与证书解析扩展位必须按其能力预留

### 5.3 可作为参考实现，但不进入核心

#### 7. `github.com/go-acme/lego`

用途：

- ACME 流程、DNS provider、联调脚本参考

选型结论：

- 仅建议作为参考，不建议进入核心运行时

原因：

- 适合快速打通一期
- 但系统长期目标是 `标准 ACME + 私有扩展字段/流程 + SM2`
- 若直接把 `lego` 放到核心，后续在 `JWS / finalize / 私有扩展流程` 上会被外部实现方式绑住

建议用法：

- 作为联调工具
- 作为阿里云 DNS 与 ACME 标准流程对照实现
- 作为协议兼容性测试基线

#### 8. `github.com/certimate-go/certimate`

用途：

- 产品实现参考、连接器能力参考

选型结论：

- 仅建议学习参考，不建议 fork 为系统底座

原因：

- 它是完整产品，不是聚焦型库
- 你当前需求是多租户 SaaS 控制面 + 客户侧多 Agent + 可演化协议子系统
- 直接以其为基础，后续拆分多租户、审计、职责分离、ACME 私有扩展会更重

### 5.4 不建议进入一期核心路径的组件

#### 9. `github.com/caddyserver/certmagic`

结论：

- 不建议进入核心

原因：

- 其设计中心是“自动取证与续期”
- 并非面向多租户证书治理平台
- 更适合作为自动化 ACME 工具，而不是你的协议内核

#### 10. `github.com/mholt/acmez`

结论：

- 不建议进入核心

原因：

- 适合作为 ACME 参考实现
- 但长期仍会与私有扩展 ACME 子系统产生职责重叠

#### 11. `github.com/hibiken/asynq`

结论：

- 可选，不建议成为业务状态机承载核心

原因：

- 可以承担作业投递和重试
- 但不能承载证书生命周期状态机本身
- 若未来任务系统需更强的一致性和审计绑定，仍要以数据库中的自有任务模型为主

## 6. 一期实施建议

### 6.1 一期建议实际落地组合

- 核心域模型：自研
- ACME 协议层：自研封装，内部基于 `x/crypto/acme`
- DNS 抽象：`libdns`
- 阿里云 DNS：`libdns/alidns`
- NGINX 发现：`gonginx`
- Tomcat PKCS12：`go-pkcs12`
- Agent / 调度 / 审计 / RBAC：自研

### 6.2 一期不要做的事

- 不要把 `lego` 直接作为 ACME 核心
- 不要把阿里云 DNS 调用直接散落在业务逻辑里
- 不要在业务层直接依赖 `rsa` 具体类型
- 不要把 NGINX/Tomcat 连接器写成无抽象的一次性脚本
- 不要把发现模型和部署模型混成一个任务类型

## 7. 面向 SM2 与私有 ACME 扩展的预留点

### 7.1 ACME 扩展预留

内部 `ACME` 协议层从一期开始就必须允许：

- 自定义 JWS Header
- 自定义 JWK / Key Type
- 自定义 order 扩展字段
- 自定义 challenge 类型
- 自定义 finalize 参数
- 自定义证书下载后处理

这意味着 `pkg/protocol/acme` 不能只是简单调用第三方库，而必须提供自己的 request/response 映射与扩展 hook。

### 7.2 证书和密钥抽象预留

从一期开始建议统一定义：

- `PrivateKeyHandle`
- `CSRHandle`
- `CertificateBundle`
- `KeySpec`
- `CertificateSpec`

至少包含：

- 算法类型
- 位数/曲线/参数
- 编码格式
- 是否允许导出
- 是否允许复用

这样后续加入 `SM2` 时，不需要修改申请单模型和部署任务模型。

### 7.3 TargetConnector 预留

连接器不能假设证书一定是：

- RSA
- X.509 标准链路
- PEM + KEY 双文件

需要预留：

- 不同算法证书链写入
- 不同 keystore 编码
- 不同私钥可导出策略

## 8. 风险与控制

### 8.1 主要风险

- ACME 私有扩展过深，导致与公有 CA 的兼容性回归
- `SM2` 与标准 Web PKI 流程存在现实兼容差异
- NGINX/Tomcat 发现与部署的配置差异比预期更大
- 许可证策略若调整不及时，会限制可引入组件范围

### 8.2 控制建议

- 所有第三方库都必须限制在适配层，不得扩散到业务层
- 每类驱动建立契约测试，保证可替换
- 为标准 ACME 与私有扩展 ACME 分别建立测试集
- 将算法能力、挑战能力、部署目标能力都表达为 `Capability` 元数据

## 9. 最终结论

本项目应坚持以下选型策略：

- 核心域模型自研
- ACME 作为内部协议子系统自研封装
- 一期以 `x/crypto/acme + libdns + alidns + gonginx + go-pkcs12` 为主要复用组件
- 后续以 `gmsm` 承接 `SM2` 扩展
- `lego`、`certimate`、`CertMagic`、`acmez` 作为参考与对照，不进入核心架构

这是当前最能兼顾以下目标的路线：

- 一期可落地
- 后续支持 `SM2`
- 支持标准 `ACME + 私有扩展字段/流程`
- 不在二期或三期更换核心组件
