# AutoCertX 后端代码结构评估与重构建议

- 编写日期：2026-04-20
- 适用范围：控制面后端、Agent 目录规划、控制台聚合查询层设计
- 关联输入：
  - `doc/一期GA详细设计.md`
  - `doc/前端页面设计.md`
  - `web/console/*.html` 原型页面

## 1. 评估目标

本评估不是判断“当前目录有没有建出来”，而是判断：

- 当前后端代码结构是否能自然承接前端页面原型
- 当前目录边界是否与详细设计一致
- 一期 GA 开发时是否会因为目录设计不当导致聚合逻辑、工作流逻辑和连接器逻辑混杂

本次评估以当前前端主导航为核心输入：

- `仪表盘`
- `域名管理`
- `证书资产`
- `CA 账户`
- `交付管理`
- `发现结果`
- `作业中心`
- `审计`
- `系统设置`

## 2. 核心结论

### 2.1 结论摘要

当前仓库的**顶层结构合理**，但**控制面内部模块结构还不够合理到可以直接作为最终业务目录展开**。

更准确地说：

- `cmd / internal / pkg / api / sql / web` 这一层没有问题
- `internal/platform` 作为基础能力层没有问题
- `pkg/protocol/acme` 单独保留也没有问题
- 问题集中在 `internal/controlplane` 当前仍然是“模块名清单”，但详细设计和前端原型已经要求控制面必须具备：
  - 明确的分层结构
  - 读写分离的查询组织方式
  - 面向页面聚合的 Query 层
  - 更稳定的 bounded context 边界

### 2.2 是否合理

判断结果：

- 仓库级结构：`合理`
- 控制面当前 README 里的模块清单：`部分合理，但不建议直接照此展开`
- Agent 结构方向：`基本合理，但缺少 challenge 和 verify 的显式落点`

## 3. 现状问题

### 3.1 控制面存在两套结构口径

当前存在明显口径冲突：

- `internal/controlplane/README.md` 采用“业务模块平铺”思路：
  - `identity`
  - `domainmgr`
  - `issuer`
  - `certificate`
  - `workflow`
  - `scheduler`
  - `job`
  - `agenthub`
  - `deployment`
  - `discovery`
  - `audit`
  - `evidence`
  - `notification`
  - `settings`
  - `stats`
- 而详细设计第 24 章已经明确建议：
  - `internal/app/controlplane`
  - `internal/domain`
  - `internal/application`
  - `internal/repository`
  - `internal/workflow`
  - `internal/driver`
  - `internal/scheduler`

这两种组织方式不能同时成立。

### 3.2 当前结构无法自然承接前端聚合页面

前端原型不是纯 CRUD 页面，而是典型的“治理 + 运行 + 聚合视图”控制台。

典型页面包括：

- `仪表盘`
- `证书资产`
- `交付管理`
- `作业中心`
- `审计`

这些页面都依赖跨域聚合数据，而不是单一实体表的 CRUD。

详细设计实际上已经定义了这类能力：

- `DetailQueryService`
- `StatsQueryService`

但当前目录结构没有为这两类能力提供清晰落点。

### 3.3 当前模块命名混杂了三种语义

现有控制面模块名同时混入了：

- 领域对象语义
  - `issuer`
  - `job`
  - `discovery`
- 应用服务或管理器语义
  - `domainmgr`
  - `agenthub`
- 查询或展示语义
  - `stats`

这种命名方式一旦直接开始编码，后续会出现：

- 聚合查询逻辑散落到领域模块
- 页面读模型和命令逻辑耦合
- “某个目录到底放 domain、application 还是 query 代码”变得不清楚

### 3.4 `交付管理` 只能是页面聚合域，不能变成后端合并域

前端里 `部署目标` 和 `节点管理` 已经合并为 `交付管理` 导航域，这是页面层收敛。

但后端仍然必须坚持：

- `DeploymentTarget`
- `AgentNode`

两个域对象分开。

也就是说：

- 导航可以合并
- 查询页可以聚合
- 领域对象和持久化模型不能合并

### 3.5 `certificate` 作为单一模块过宽

从 UI 原型和详细设计看，证书相关至少分成三块：

- `CertificateAsset`
- `CertificateRequest`
- `IssueWorkflow`

如果全部堆入单一 `certificate` 模块：

- 资产治理
- 申请创建
- 工作流推进
- 部署发现聚合

会迅速混在一起。

## 4. 推荐的控制面代码组织

## 4.1 总体结构

建议将控制面正式收敛为如下结构：

```text
internal/
  app/
    controlplane/
      bootstrap/
      http/
      middleware/
      wiring/
  domain/
    identity/
    tenancy/
    domains/
    dnscredentials/
    issuer/
    certificateasset/
    certificaterequest/
    issueworkflow/
    deploymenttarget/
    agentnode/
    discovery/
    job/
    audit/
    settings/
  application/
    command/
      domains/
      caaccounts/
      assets/
      requests/
      renewals/
      targets/
      nodes/
      discoveries/
      jobs/
      settings/
    query/
      dashboard/
      domains/
      assets/
      delivery/
      discoveries/
      jobs/
      audit/
      settings/
      authcontext/
  repository/
  workflow/
  driver/
  scheduler/
```

### 4.2 分层职责

`internal/app/controlplane`
- 负责进程启动、依赖装配、HTTP 路由、中间件、生命周期管理
- 不承载业务规则

`internal/domain`
- 承载领域对象、领域规则、状态机和聚合根
- 保持和页面无关

`internal/application/command`
- 承载写入型用例
- 例如：
  - 创建域名
  - 创建证书申请
  - 手动续期
  - 停用节点
  - 忽略发现

`internal/application/query`
- 承载面向控制台页面的聚合查询
- 这是当前最需要补的一层

`internal/repository`
- 持久化抽象和具体实现
- 负责表级读写，不负责页面组装

`internal/workflow`
- 签发、挑战、部署、续期等编排逻辑
- 不直接承担 HTTP 展示职责

`internal/driver`
- 外部系统适配
- 包括：
  - ACME
  - AliDNS
  - Agent transport
  - 部署连接器驱动

`internal/scheduler`
- `jobs + job_attempts + SKIP LOCKED + lease` 主调度机制
- 与页面层、HTTP 层隔离

## 6. handler / service / repository 拆分建议

## 6.1 推荐原则

不建议使用“大 service”承接整页逻辑，而应按以下方式拆分：

- `handler`
  - 只处理 HTTP 输入输出、鉴权、参数校验、错误映射
- `application command/query`
  - 处理用例和聚合
- `repository`
  - 处理表和存储细节

## 6.2 页面映射表示例

| 页面 / API 域 | Handler 建议 | Query / Command 建议 | Repository 关注点 |
| --- | --- | --- | --- |
| 仪表盘 | `dashboard_handler.go` | `dashboard.SummaryQuery` | 资产、作业、发现、节点读模型 |
| 域名管理 | `domains_handler.go` | `domains.ListQuery` `domains.UpsertCommand` | `domain_assets` `domain_validation_records` `dns_record_operations` |
| 证书资产 | `assets_handler.go` | `assets.ListQuery` `assets.DetailQuery` `requests.CreateCommand` | `certificate_assets` `certificate_versions` `certificate_requests` |
| CA 账户 | `ca_accounts_handler.go` | `caaccounts.ListQuery` `caaccounts.UpsertCommand` | `ca_accounts` `acme_accounts` |
| 交付管理 | `delivery_handler.go` | `delivery.WorkspaceQuery` | `deployment_targets` `agents` |
| 发现结果 | `discoveries_handler.go` | `discoveries.ListQuery` `discoveries.ClaimCommand` | `discovery_records` |
| 作业中心 | `jobs_handler.go` | `jobs.ListQuery` `jobs.DetailQuery` `jobs.RetryCommand` | `jobs` `job_attempts` |
| 审计 | `audit_handler.go` | `audit.ListQuery` `audit.DetailQuery` | `audit_events` |
| 系统设置 | `settings_handler.go` | `settings.GetQuery` `settings.UpdateCommand` | `system_settings` `webhook_endpoints` |

## 7. Agent 结构建议

当前 Agent 结构大方向合理，但建议补齐两个显式能力层：

```text
internal/agent/
  bootstrap/
  poller/
  executor/
  keymgr/
  challenge/
    http01/
  deploy/
    nginx/
    tomcat/
  verify/
    nginx/
    tomcat/
  discover/
    nginx/
    tomcat/
  evidence/
  reporter/
```

原因：

- 一期需求里 `HTTP-01` 是独立能力
- 部署后验证在产品里是一级能力，不应只作为 deploy 内部隐含步骤

## 8. 对当前仓库的具体调整建议

### 8.1 应立即调整

1. 将 `internal/controlplane/README.md` 改为分层结构说明，不再使用旧模块平铺口径。
2. 在 README 中同步修正控制面目录职责，避免出现两套描述。
3. 后续真实落代码时，优先创建：
   - `internal/app/controlplane`
   - `internal/domain`
   - `internal/application`
   - `internal/repository`
4. 不要把 `交付管理` 直接实现成一个统一后端模块。

### 8.2 可后置调整

1. Agent 侧补 `challenge/http01`
2. Agent 侧补 `verify/*`
3. 再根据 OpenAPI 冻结结果细化 handler 目录

## 9. 最终建议

最终建议不是“大规模重构当前仓库”，而是：

- 保留当前顶层仓库结构
- 停止按 `internal/controlplane/README.md` 里的旧模块名继续扩展
- 以详细设计第 24 章为准，正式切到：
  - `app`
  - `domain`
  - `application`
  - `repository`
  - `workflow`
  - `driver`
  - `scheduler`

一句话概括：

**前端页面已经说明这个系统不是 CRUD 后台，而是强聚合、强编排的治理控制台；后端必须从一开始为 `query 聚合层` 和 `稳定领域边界` 预留位置，否则后续页面一落地，目录结构会很快失控。**
