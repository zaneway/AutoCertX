- 文档版本：V1.1（基于头脑风暴方案补充完善）

- 最后更新：2026-04-13
- 产品定位：多租户 SaaS 控制面 + 客户侧专属执行面（Execution Plane）的企业证书生命周期平台

## 1. 背景

企业证书管理正在从"低频人工运维"进入"高频自动化编排"阶段，驱动因素包括：

- 公网 TLS 证书生命周期持续缩短（Let's Encrypt 已缩短至 6 天短期证书，行业趋势向更短有效期演进），人工续期与人工分发不可持续。
- 企业基础设施异构，证书分布在 NGINX、Apache Tomcat、HAProxy、Kubernetes、传统 Linux 主机等多类目标端。
- 企业同时存在公网 CA（Let's Encrypt、DigiCert、Sectigo 等）与内网私有 CA（ADCS、step-ca、Vault PKI 等），证书申请、安装、续期、吊销、审计分散在多个工具链中，缺乏统一管控面。
- 商业客户不仅要求自动化，还要求多租户隔离、审批流、审计留痕、证据导出与职责分离。

本项目目标是构建一个可对外提供托管/商用能力的证书生命周期平台。平台托管控制面，客户部署专属执行面；私钥始终保留在客户边界内，平台只负责编排、策略、审计和资产可视化。

### 1.1 竞品与行业现状

当前行业主要方案如下：

| 方案                  | 类型    | 优势                          | 劣势           | 适用场景          |
| ------------------- | ----- | --------------------------- | ------------ | ------------- |
| Venafi              | 商业    | 功能最全面、企业级治理、TPP 协议          | 价格昂贵、闭源、部署重  | 大型企业          |
| HashiCorp Vault     | 开源+商业 | 密钥管理生态、PKI 引擎、动态密钥          | 非证书专用、学习曲线陡  | 已有 Vault 的企业  |
| cert-manager        | 开源    | Kubernetes 原生、CRD 模型、社区活跃   | 仅限 K8s、无 GUI | Kubernetes 环境 |
| Smallstep (step-ca) | 开源+商业 | 轻量级私有 CA、SSH 证书、ACME Server | 非全生命周期管理     | 内网私有 CA       |
| Keyfactor           | 商业    | 企业级 CLM、发现能力、合规报告           | 价格昂贵         | 金融/医疗行业       |
| certctl（参考项目）       | 开源    | 全栈自托管、Agent 架构、连接器模型        | 企业治理能力不足     | 中小企业          |

**我们的市场定位**：面向中大型企业的、开源可自托管的企业级证书生命周期管理平台，填补 cert-manager（仅 K8s）与 Venafi（昂贵商业）之间的市场空白。

### 1.2 参考项目：certctl

certctl 是一个开源的自托管证书生命周期管理平台（[GitHub: shankar0123/certctl](https://github.com/shankar0123/certctl)），采用控制平面 + Agent 分离架构，在架构设计上为本项目提供了重要参考：

- **控制平面**：Go HTTP 服务器 + PostgreSQL，提供 REST API（55+ 端点）和 Web Dashboard（React SPA）
- **Agent**：轻量级 Go 进程，部署在基础设施端，负责密钥生成、CSR 提交、证书部署
- **连接器架构**：可插拔的 Issuer/Target/Notifier 连接器
- **安全模型**：私钥隔离（Agent 本地生成 ECDSA P-256 密钥，仅提交 CSR）、Pull-Only 部署

certctl 的核心能力覆盖了 CA 后端（Local CA、ACME v2、step-ca、Vault PKI、DigiCert 等）、部署目标（NGINX、Apache、HAProxy、F5、IIS 等 10+ 种）、通知渠道（Email、Webhook、Slack 等）、证书发现扫描、网络扫描、CRL/OCSP、EST Server (RFC 7030)、ARI (RFC 9773) 等。

certctl 的不足与本项目的差异化方向详见第 20 节"与参考项目（certctl）的差异化分析"。

## 2. 项目目标

### 2.1 业务目标

- 为中大型企业提供统一的证书生命周期平台，覆盖申请、审批、签发、分发、安装、热更新、续期、吊销与审计。
- 通过 SaaS 模式降低客户部署和运维成本，同时通过客户侧执行面控制信任边界。
- 形成可商用的分层能力：多租户、套餐、租户管理、审批治理、合规证据、开放 API。
- 支持抽象钉钉/飞书/企业微信等接口，便于对接通知、等保 2.0 合规模板。
- 具备完善的多租户和工作流机制，并具备对应的合规审计记录。
- 提供证书资产全景可视化，包括证书分布拓扑、到期预测、合规状态仪表板。

### 2.2 技术目标

- 支持混合基础设施，不绑定单一运行环境或单一 CA。
- 私钥默认不进入平台控制面，满足更清晰的责任边界与更低的数据托管风险。
- 控制面从一开始按分布式、高可用、可审计、可扩展架构设计，不沿用单机调度模型。
- 支持连接器可插拔架构，Issuer/Target/Notifier 三类连接器均可扩展。
- 支持国际化（中英双语），可扩展多语言。

## 3. 非目标

- V1 不覆盖 IoT/SCEP/SCEP Proxy、移动设备证书、终端设备注册等设备身份场景。
- V1 不自建公有 CA，不提供浏览器信任服务。
- V1 不支持平台侧统一托管客户私钥。
- V1 不追求大而全连接器矩阵，优先覆盖企业高频场景。
- V1 不提供本地化私有部署版本；产品形态默认为托管式 SaaS。
- V1 不内置 ACME Server 模式（平台自身作为 ACME Server），留作 V2 候选。
- V1 不内置 EST Server (RFC 7030)，留作 V2 候选。
- V1 不内置 AI 辅助能力（证书异常检测、续期策略优化建议），留作远期规划。

## 4. 产品定位与部署模型

### 4.1 产品定位

本产品是"企业混合基础设施证书生命周期平台"，面向如下客户：

- 同时管理公网 TLS 与内网 TLS 证书的中大型企业
- 需要统一纳管 NGINX、Apache Tomcat、HAProxy、Kubernetes、Linux 主机等基础设施
- 对审批、职责分离、审计导出、证据留存有明确要求的安全/平台团队

### 4.2 部署模型

- 平台控制面：由服务提供方统一托管，多租户 SaaS。
- 客户执行面：客户侧部署专属 Agent / Relay / Proxy 集群。
- 网络方向：执行面主动连接控制面拉取任务；控制面不主动打入客户网络。
- 密钥边界：私钥仅在客户执行面或客户指定密钥设施内生成、使用、存储。

### 4.3 技术选型

| 层次     | 技术选型                                | 理由                          |
| ------ | ----------------------------------- | --------------------------- |
| 控制平面后端 | Go 1.22+                            | 高性能、静态编译、crypto 库生态丰富       |
| Web 前端 | React + TypeScript + TanStack Query | 生态成熟、服务端状态管理优秀              |
| 主数据库   | PostgreSQL 16                       | 成熟可靠、JSONB 支持、LISTEN/NOTIFY |
| 缓存     | Redis                               | 会话管理、限流、任务队列、热数据缓存          |
| Agent  | Go（与控制平面共享代码库）                      | 代码复用、单一二进制部署                |
| 容器化    | Docker + Kubernetes                 | 标准化部署、弹性伸缩                  |
| CI/CD  | GitHub Actions                      | 标准化流程                       |
| API 文档 | OpenAPI 3.0                         | 自动生成文档和 SDK                 |
| 认证     | JWT + OIDC/SAML                     | 企业 SSO 集成                   |

## 5. 产品原则

- 控制面托管，执行面下沉。
- 私钥永不进入 SaaS 控制面。
- 所有副作用操作必须幂等。
- 所有后台编排必须支持分布式 claim/lease，不能依赖单实例调度器。
- 合规与审计能力优先于连接器数量。
- 先打通高频企业场景，再扩展长尾连接器。
- 连接器可插拔：Issuer/Target/Notifier 均通过接口抽象，支持自定义扩展。
- 安全优先：零信任架构、端到端加密、最小权限原则。

## 6. 用户角色

### 6.1 平台侧角色

- 平台超级管理员：管理租户、套餐、系统级审计策略、平台级配置。
- 平台运营人员：管理租户生命周期、配额、工单、通知模板。

### 6.2 租户侧角色

- 租户管理员：管理组织、环境、用户、Execution Plane、Issuer、Target、项目级配置。
- 安全管理员：管理审批策略、证书模板、吊销策略、例外审批、审计导出。
- 平台工程师：负责证书资产、目标端映射、自动续期、部署策略与故障处理。
- 审计员：只读查看审计、审批、证据、报表。
- 执行面运维人员：负责客户侧执行面部署、升级、节点健康与网络打通。

### 6.3 角色职责分离要求

- 租户管理员不能默认绕过安全审批策略。
- 安全管理员与平台工程师的权限必须可独立配置，支持职责分离。
- 发起人不能审批自己的高风险请求。
- 审计员仅具备只读访问权，无法触发任何资源变更。
- 高风险操作（吊销、策略例外、敏感模板修改）必须支持双人审批或双角色隔离。

## 8. 关键术语

- 控制面：SaaS 托管的管理与编排平面，负责租户、策略、工作流、审计、资产台账、开放接口。
- 执行面：部署在客户环境的节点集群，负责私钥生成、调用客户侧凭据、部署证书、执行 reload、回传结果。
- Issuer：证书签发源，如 ACME、Vault PKI。
- Target：证书部署目标，如 NGINX、Kubernetes Secret、SSH 远程主机。
- 证书资产：被平台纳管的证书对象及其生命周期、所有权、历史版本、部署历史。
- 证据：可用于审计或客户证明的结构化记录，如审批链、部署记录、验证结果、操作责任人。
- 连接器（Connector）：可插拔的集成模块，分为 Issuer Connector（CA 签发）、Target Connector（目标部署）、Notifier Connector（通知发送）三类。
- 证书模板（Certificate Profile）：预定义的证书参数模板，包括密钥类型/长度、有效期上限、SAN 规则、允许 Issuer、允许目标端类型等。
- 策略引擎（Policy Engine）：基于规则的证书合规校验系统，支持策略继承与覆盖。

## 9. 总体能力模型

平台分为五个能力域：

1. **租户治理域**：组织、环境、身份、权限、审批、审计、配额
2. **证书资产域**：证书申请、版本、到期、部署关系、吊销状态
3. **编排执行域**：工作流、任务队列、Execution Plane、目标端连接器
4. **集成连接域**：Issuer Connector、Target Connector、通知与 API
5. **合规证据域**：操作审计、审批留痕、部署证据、报表与导出

## 10. 功能需求

### 10.1 多租户与组织管理

#### 需求

- 系统必须支持多租户模型。
- 每个租户必须拥有独立的组织空间、用户空间、资源命名空间和审计视图。
- 资源对象必须带租户 ID，核心查询必须默认带租户边界。
- 必须支持环境分层，如 `production / staging / dev`。
- 必须支持项目、业务线或应用分组，作为证书和目标端归属维度。
- 必须支持租户配额与套餐限制，如证书数量、Execution Plane 节点数、API 调用量。
- 必须支持组织级（Organization）→ 项目级（Project）→ 环境级（Environment）三级隔离。

#### 验收标准

- 任意租户无法查看、检索、导出其他租户资源。
- 配额超限时，系统能明确拒绝资源创建并给出可观测错误。
- 同一租户内，不同项目/环境的资源可独立管理。

### 10.2 身份认证与权限控制

#### 需求

- 必须支持 OIDC 和 SAML 2.0 SSO。
- 必须支持本地紧急管理员账户，但默认禁用或受限。
- 必须支持细粒度 RBAC，至少覆盖租户、环境、资源类型、操作级权限。
- 必须支持服务账号与 API Token。
- 必须支持基于角色的职责分离。
- 必须支持 API Key + JWT 双模式认证。

#### 强合规要求

- 高风险操作必须支持双人审批或双角色隔离。
- 租户管理员不能默认绕过安全审批策略。
- 权限变更必须记录到审计系统。

#### 验收标准

- 安全管理员可禁止平台工程师直接执行吊销、策略例外、敏感模板修改。
- 审计员具备只读访问权，无法触发资源变更。

### 10.3 Issuer 管理

#### 需求

- 系统必须支持 ACME v2 连接器，支持 HTTP-01 和 DNS-01 验证方式。
- 系统必须支持一个企业私有 CA 连接器，V1 默认假设为 Vault PKI。
- 每个 Issuer 的凭据和配置必须按租户隔离，并使用平台侧密钥管理机制加密存储。
- 必须支持 Issuer 连接性测试。
- 必须支持 Issuer 与环境/项目/模板之间的绑定关系。
- 必须支持不同证书模板选择不同 Issuer。
- 必须支持多 CA 冗余与自动故障切换（当主 CA 不可用时自动切换到备用 CA）。

#### 验收标准

- 同一租户可并存公网 CA 与私有 CA。
- 不同租户之间的 Issuer 配置完全隔离。
- Issuer 连接性测试能准确反映 CA 可用状态。

### 10.4 Execution Plane 管理

#### 需求

- 客户必须能够注册多个执行面节点。
- 执行面节点必须支持分组、区域标签、网络域标签、用途标签。
- 节点必须支持心跳、版本上报、健康状态、最后活跃时间。
- 节点必须上报系统元数据（OS、架构、IP 地址、主机名、版本）。
- 必须支持滚动升级与版本兼容检查。
- 必须支持任务拉取、断线恢复、失败重试。
- 执行面必须是单一二进制文件，无外部依赖。
- 执行面必须支持 mTLS 双向认证与控制面通信。

#### 约束

- 控制面不得主动连入执行面。
- 执行面必须是任务执行、私钥使用、目标凭据使用的唯一责任点。

#### 验收标准

- 执行面短时离线后恢复时，可继续处理未过期任务。
- 无健康执行面时，系统能够阻止高风险部署任务进入不可恢复状态。

### 10.5 证书模板与策略

#### 需求

- 必须支持证书模板，定义常用参数：CN、SAN 规则、用途、有效期上限、允许 Issuer、允许目标端类型。
- 必须支持按环境、项目、系统类别绑定模板。
- 必须支持自动续期窗口、到期阈值、审批要求、告警规则。
- 必须支持策略例外申请与审批。
- 策略引擎必须支持策略继承与覆盖：组织级默认策略 → 项目级覆盖 → 环境级差异化。
- 策略引擎必须支持实时违规检测：证书申请时自动校验策略合规性。
- 策略维度必须覆盖：密钥类型/长度、证书有效期、SAN 限制、CA 白名单、自动续期窗口、吊销策略。

#### 强合规要求

- 模板修改必须审计。
- 高风险模板必须限制仅安全管理员可改。
- 策略违规必须产生告警并记录到审计系统。

### 10.6 证书申请与签发

#### 需求

- 用户可基于模板创建证书申请。
- 申请必须支持填写或选择 CN、SAN、环境、项目、部署目标、所有权人。
- 系统必须支持审批前置。
- 审批通过后，系统创建签发工作流。
- 在客户侧持钥模式下，私钥必须由执行面生成，控制面只接收 CSR。
- 执行面必须支持多种密钥算法：ECDSA P-256、RSA-2048、Ed25519。
- 签发工作流必须支持异步签发模式（适用于 DigiCert 等需要轮询订单状态的 CA）。

#### 验收标准

- 控制面数据库、日志、审计中不得出现私钥材料。
- 申请记录、审批记录、签发记录必须可追溯。

### 10.7 证书资产台账

#### 需求

- 必须支持证书资产台账列表、搜索、过滤、标签、批量操作。
- 必须展示证书状态、到期时间、关联 Issuer、关联目标端、关联执行面、最近部署结果。
- 必须支持证书版本历史。
- 必须支持查看历史审批、历史部署、历史吊销记录。
- 必须支持证书到期预测：基于历史数据预测未来 90 天的证书到期风险。
- 必须支持证书链追溯：从证书到签发 CA 到根 CA 的完整信任链可视化。
- 必须支持资产全景图：可视化展示证书分布拓扑。

#### 验收标准

- 任意一张证书的"谁申请、谁审批、什么时候部署到哪里、部署结果如何"必须能单跳查询到。

### 10.8 自动续期

#### 需求

- 必须支持自动续期策略。
- 必须支持续期窗口配置、续期前审批、失败重试、失败告警。
- 必须支持基于 Issuer 能力的续期策略扩展，如 ACME ARI (RFC 9773)。
- 续期流程必须复用客户侧持钥模式。
- 续期调度器必须基于分布式 claim/lease 机制，确保在控制面多副本运行时不会产生重复续期。

#### 架构要求

- 续期编排必须基于分布式任务系统。
- 同一证书同一轮续期必须具备幂等键。
- 系统必须防止多副本控制面重复创建续期任务。

#### 验收标准

- 在控制面多副本运行时，同一证书不会因并发调度产生重复续期。

### 10.9 分发、安装与热更新

#### 需求

- 必须支持将签发后的证书分发到绑定目标端。
- 每个目标端连接器必须至少定义：
  - 配置校验
  - 安装逻辑
  - reload/reconfigure 逻辑
  - 安装后验证逻辑
  - 幂等要求
- 系统必须定义统一的热更新能力等级：
  - `HotReload`：无需连接中断即可生效
  - `GracefulReload`：平滑重载
  - `RestartRequired`：需重启服务
- 新证书部署失败时，系统必须支持自动回滚到旧证书（优雅回滚）。

#### 目标端 V1 验收

- NGINX：支持文件写入、配置校验、reload、可选 TLS 回探验证
- Apache：支持文件写入、配置校验、graceful reload
- HAProxy：支持组合 PEM、配置校验、reload
- Kubernetes Secret：支持 Secret 更新与版本标记，是否热更新由消费方能力决定
- SSH：支持通过客户执行面远程分发证书到 Linux 主机

### 10.10 部署后验证

#### 需求

- 每次部署完成后，系统必须记录部署结果。
- 必须支持安装级验证，如文件存在、权限、命令退出码。
- 必须支持服务级验证，如 reload 是否成功。
- 必须支持可选网络级验证，如 TLS 指纹回探（TLS 握手验证）。
- 验证结果必须形成结构化证据。

#### 验收标准

- 平台可导出"某次部署完成且服务端实际提供了对应证书"的证据记录。

### 10.11 吊销与紧急处置

#### 需求

- 必须支持手工吊销。
- 必须支持紧急替换场景：新证书签发、批量分发、旧证书标记失效。
- 必须支持吊销原因记录。
- 吊销操作必须支持高风险审批或双人复核。
- 吊销操作必须通知对应 Issuer 执行吊销（如 ACME、Vault PKI）。

#### 验收标准

- 吊销与替换过程均有完整审计链和结果状态。

### 10.12 审批流与例外管理

#### 需求

- 必须支持基于动作类型的审批流，如：
  - 新证书申请审批
  - 高风险模板审批
  - 续期审批
  - 吊销审批
  - 策略例外审批
- 必须支持多级审批：串行审批、并行审批、会签、或签。
- 必须支持审批条件路由：按证书类型、域名、环境等条件自动路由到不同审批链。
- 必须支持双人审批。
- 必须支持审批超时自动处理（升级或拒绝）。
- 必须支持审批拒绝理由和审计。
- 审批待办必须支持推送到企业 IM（钉钉、飞书、企业微信）。
- 审批链记录必须不可篡改。

#### 强合规要求

- 审批链必须可导出。
- 发起人不能审批自己的高风险请求。

### 10.13 审计、证据与报表

#### 需求

- 必须记录所有安全相关和资源相关操作。
- 审计记录必须包含：
  - 谁发起
  - 谁审批
  - 谁执行
  - 在哪个租户/环境
  - 作用于哪个资源
  - 结果如何
  - 关联任务 ID / 工作流 ID
- 必须支持证据导出，格式包括 PDF、CSV、JSON。
- 必须支持按时间范围、资源范围、角色范围导出报表。
- 必须支持证书到期预测报表。

#### 强合规要求

- 审计系统必须具备防篡改设计，建议采用 hash chain 与 WORM/Object Lock 存储归档。
- 审计导出必须具备唯一导出编号和导出审计。
- 审计日志应支持使用平台 CA 进行数字签名，防止日志篡改。

### 10.14 通知与集成

#### 需求

- 必须支持邮件和 Webhook。
- V1 建议支持 Slack 类通知。
- 通知必须支持事件分级、去重、抑制、升级。
- 必须支持外部系统订阅关键事件，如审批完成、部署失败、即将到期。
- 必须支持分级告警：证书到期 90/60/30/14/7/1 天多级告警。
- 必须支持告警升级：未确认告警自动升级到上级。
- 必须支持告警静默与抑制：维护窗口期告警静默、重复告警抑制。
- 必须支持自定义通知模板。

### 10.15 开放接口

#### 需求

- 必须提供 REST API。
- API 必须带租户边界、权限校验和审计。
- API 必须提供 OpenAPI 3.0 规范文档。
- 后续可扩展 CLI/SDK，但不作为 V1 阻塞项。

### 10.16 Web 管理控制台

#### 需求

- 必须提供 Web Dashboard 作为主要操作界面。
- 必须支持以下视图：
  - 仪表板：证书统计、到期时间线、最近操作、告警概览
  - 证书资产列表：支持多选批量操作、搜索过滤
  - 证书详情：部署状态时间线、版本历史、审批记录
  - 执行面管理：节点列表、健康状态、系统信息
  - 作业队列：状态追踪、重试、取消
  - 审批中心：待办、已办、审批链查看
  - 审计日志：时间范围、角色、操作过滤，支持 CSV/JSON 导出
  - 策略管理：规则配置、违规查看
  - Issuer 管理：列表、连接测试
  - 目标端管理：配置向导
  - 组织与用户管理
- 必须支持中英双语切换。
- 必须支持响应式布局，适配桌面端。

## 11. 业务流程要求

### 11.1 申请流程

1. 用户选择模板并提交申请
2. 系统校验模板约束与权限
3. 系统校验策略引擎合规性
4. 触发审批流（如需审批）
5. 审批通过后创建签发任务
6. 执行面生成私钥与 CSR
7. 控制面调用 Issuer 完成签发
8. 控制面创建部署任务
9. 执行面完成安装、reload、验证并回传证据
10. 系统记录审计日志

### 11.2 自动续期流程

1. 控制面根据续期窗口扫描资产
2. 生成带幂等键的续期任务（基于分布式 claim/lease）
3. 如需审批则等待审批
4. 执行面重新生成密钥或按策略复用
5. 控制面完成续期签发
6. 执行面分发并安装
7. 执行面执行部署后验证
8. 验证失败时自动回滚到旧证书
9. 系统记录验证结果与审计

### 11.3 紧急替换流程

1. 安全管理员发起紧急替换
2. 双人审批
3. 优先签发替代证书
4. 批量部署到绑定目标端
5. 验证成功后可选吊销旧证书
6. 通知 Issuer 执行吊销
7. 导出处置证据

### 11.4 证书发现流程（V2 候选）

1. 执行面扫描本地文件系统中的证书文件
2. 网络扫描器对指定 IP/端口范围进行 TLS 握手
3. Kubernetes 扫描器发现集群中的证书 Secret 和 Ingress TLS 配置
4. 系统基于 SHA-256 指纹去重
5. 未纳管证书进入待认领列表
6. 用户认领或忽略发现的证书
7. 认领后纳入平台管理

## 12. 非功能需求

### 12.1 可用性

- 控制面可用性目标不低于 99.9%
- 核心工作流组件不得单点
- 审计与任务系统不得共享单点故障
- 控制面必须支持水平扩展（无状态设计）
- 调度器必须使用分布式锁（Leader Election）确保单 Leader

### 12.2 安全性

- 全链路 TLS
- 平台侧 secrets 必须加密存储
- 每租户独立加密域或至少逻辑分域
- 所有访问必须鉴权与授权
- 敏感配置读取必须审计
- 执行面与控制面之间必须使用 mTLS 双向认证
- Agent 二进制必须支持签名验证，防止篡改

### 12.3 隔离性

- 租户数据必须逻辑隔离
- 执行面路由必须按租户绑定
- 不允许跨租户执行任务

### 12.4 可扩展性

- 控制面必须支持横向扩展
- 工作流/任务系统必须支持高并发任务处理
- Connector 设计必须支持新增 Issuer/Target 类型
- V1 性能目标：支持 10K+ 证书、1K+ 执行面节点

### 12.5 可观测性

- 必须具备 metrics、日志、trace
- 必须具备租户级和平台级任务视图
- 必须具备执行面健康监控
- 必须具备证书到期时间线仪表板

### 12.6 可运维性

- 支持灰度发布
- 支持特性开关
- 支持租户级限流、配额与熔断
- 支持执行面版本兼容矩阵
- 支持数据库主从复制与连接池

### 12.7 数据存储

- 主数据库：PostgreSQL（证书元数据、租户、策略、审计日志）
- 缓存层：Redis（会话、API 限流、热数据缓存、任务队列）
- 审计库：可选独立审计库，append-only 存储
- 任务分发：PostgreSQL LISTEN/NOTIFY 或 Redis Streams

## 13. 数据与安全边界要求

### 13.1 平台控制面允许存储的数据

- 证书元数据
- CSR
- 已签发证书和证书链
- 目标端定义
- Issuer 配置
- 审批记录
- 审计记录
- 部署结果和验证证据

### 13.2 平台控制面禁止存储的数据

- 客户私钥明文
- 执行面本地私钥文件
- 非必要的目标端高权限凭据明文

### 13.3 客户执行面负责的数据

- 私钥生成与保存（本地加密存储，文件权限 0600）
- 目标端凭据的本地使用
- 本地部署上下文
- 本地 reload 执行
- 离线期间的证书本地缓存

## 14. 架构约束

- 不允许采用"每个 API 实例各自跑调度循环、通过内存防重"的模式。
- 必须采用可分布式扩展的编排方式，例如：
  - 任务表 + claim/lease 机制
  - 专用任务队列
  - Workflow Engine
- 所有可重试任务必须定义：
  - 幂等键
  - 重试策略
  - 超时策略
  - 死信策略
  - 补偿策略
- 连接器执行必须是可审计、可观测、可限制权限的。
- 连接器必须通过接口抽象（Issuer Connector Interface、Target Connector Interface、Notifier Connector Interface），支持运行时注册和扩展。

## 15. V1 验收标准

V1 交付应满足以下最小验收：

- 支持多租户、SSO、RBAC
- 支持双人审批与策略例外
- 支持 ACME + Vault PKI
- 支持 NGINX / Apache / HAProxy / Kubernetes Secret / SSH
- 支持客户侧持钥
- 支持自动续期（分布式调度、幂等键）
- 支持部署后验证（含优雅回滚）
- 支持全量审计和证据导出
- 控制面支持多副本部署且无重复签发/重复部署问题
- 提供 Web 管理控制台（中英双语）
- 提供 OpenAPI 3.0 规范文档

## 17. 当前假设

- 平台形态为托管式 SaaS，不提供本地化部署版本。
- 客户接受部署 Execution Plane。
- 私钥默认且始终保留在客户边界。
- V1 企业私有 CA 默认假设为 Vault PKI。
- V1 不做计费系统闭环，但套餐/配额模型需预留。

## 18. 开放问题

- V1 企业私有 CA 最终是否确定为 Vault PKI，还是需保留与 step-ca 的并行验证。
- 审计防篡改的落地方案采用对象存储锁定、外部审计归档，还是专用审计存储。
- Kubernetes Secret 场景是否纳入"平台定义热更新结果"的强校验，还是仅验证 Secret 已更新。
- 商业模式上是否需要在 V1 就引入套餐、计费、账单和用量对账。

## 19. 风险与应对

| 风险         | 影响 | 应对策略                            |
| ---------- | -- | ------------------------------- |
| 私钥泄露       | 最高 | 执行面本地生成、私钥加密存储、传输仅 CSR、HSM 集成预留 |
| 证书过期导致服务中断 | 高  | 多级告警、自动续期、部署后验证、优雅回滚            |
| 控制平面单点故障   | 高  | 无状态设计、水平扩展、健康检查、Leader Election |
| 执行面被入侵     | 中  | mTLS 双向认证、最小权限、Agent 签名验证       |
| 合规要求变更     | 中  | 策略引擎可配置、合规模板热更新                 |
| CA 服务不可用   | 中  | 多 CA 冗余、自动故障切换、本地 CA 兜底         |
| 分布式调度竞争    | 中  | 分布式 claim/lease、幂等键、防重设计        |

## 20. 与参考项目（certctl）的差异化分析

基于对 certctl 项目的深入分析，本项目在以下维度进行差异化设计：

| 维度      | certctl 现状                | 本项目差异化                                    |
| ------- | ------------------------- | ----------------------------------------- |
| 产品形态    | 自托管单实例                    | 多租户 SaaS + 客户侧执行面                         |
| 多租户     | Teams/Owners 概念，无真正隔离     | 完整的租户隔离 + RBAC + 资源配额，三级隔离模型              |
| 审批流     | 无                         | 内置工作流引擎，支持多级审批、条件路由、超时处理                  |
| 合规审计    | 基础审计日志（append-only）       | 审计签名 + hash chain 防篡改 + 合规报告模板 + 证据导出     |
| 职责分离    | 无                         | 四眼原则 + 角色分离 + 操作确认 + 权限变更审计               |
| K8s 集成  | 简单 Secret 连接器             | V2 规划 cert-manager CRD 深度集成 + Operator 模式 |
| 高可用     | 单实例部署                     | 水平扩展 + Leader Election + 分布式调度            |
| 国际化     | 仅英文                       | 中英双语，可扩展多语言                               |
| 策略引擎    | 简单规则（policy\_rules）       | 条件组合 + 继承覆盖 + 合规模板 + 实时违规检测               |
| 证书发现    | Agent 扫描 + 网络扫描           | V2 规划 + K8s 集群扫描 + 云资源扫描 + 指纹去重           |
| 通知渠道    | Email/Webhook/Slack/Teams | V1 Slack + V2 钉钉/飞书/企业微信（国内企业友好）          |
| API 网关  | 无                         | 多租户路由 + 限流 + 审计日志采集                       |
| 部署回滚    | 无                         | 优雅回滚：新证书部署失败时自动回滚到旧证书                     |
| CA 冗余   | 无                         | 多 CA 冗余 + 自动故障切换                          |
| Web 控制台 | React SPA（21 页面）          | 类似丰富度 + 中英双语 + 审批中心 + 资产全景图               |
| 技术选型    | Go + PostgreSQL + React   | 一致，增加 Redis 缓存层                           |

## 21. 结论

本项目 V1 应聚焦为：

- 多租户 SaaS 控制面
- 客户侧 Execution Plane
- 客户侧持钥
- 企业高频目标端连接器
- 强治理与强合规能力优先
- 分布式高可用架构

该方向适合构建可对外提供托管/商用能力的企业证书生命周期平台，并能在后续平滑演进到更丰富的连接器矩阵和更复杂的合规场景。
